7. セキュリティ設計

7-1. 実装済みセキュリティ対策

🔐
API キーの完全サーバー隔離
ANTHROPIC_API_KEY・SUPABASE_SERVICE_ROLE_KEY・GOOGLE_MAPS_API_KEY・ADMIN_PASSWORD は NEXT_PUBLIC_ プレフィックスなし。ブラウザバンドルに含まれず、サーバーサイドのみでアクセス可能。
🗂️
Supabase クライアントの分離
src/lib/supabase.ts は anon key のみ（クライアント / サーバー共通）。src/lib/supabase-server.ts は SERVICE_ROLE_KEY を持つ admin クライアントのみ（サーバー専用 API ルートからのみ import）。誤って CLIENT コンポーネントが admin クライアントを import できない構造。
🚫
internal_notes の公開 API 除外
店舗の内部メモ（営業交渉メモ・スタッフ情報等）は GET /api/restaurants および GET /api/restaurants/[id] の SELECT 句から除外。Claude へのシステムプロンプトのみで使用し、レスポンス JSON には含まれない。
⏱️
AI チャット API レート制限
POST /api/chat に IP ベースレート制限を実装。1IP あたり 1分間 10リクエストまで。超過時は 429 Too Many Requests + Retry-After ヘッダーを返す。Anthropic API コストの暴走を防ぐ。
実装: src/lib/rate-limit.ts（インメモリ）
🔒
Admin API 認証 + ブルートフォース保護
/api/admin/spots・/api/admin/analytics・/api/admin/spots/[id] は全て Authorization: Bearer <ADMIN_PASSWORD> ヘッダーが必要。ヘッダーなし / 不正トークンは 401 Unauthorized。加えて 1IP あたり 1分 10回のレート制限でブルートフォース攻撃を防止。
実装: src/lib/admin-auth.ts
📷
Google Maps API サーバープロキシ
GET /api/photo?ref={photo_reference} が Google Places 写真を取得してブラウザに返す。Google Maps API キーはブラウザに一切送信されない。レスポンスに Cache-Control: public, max-age=86400 を付与しコストも最適化。
🙈
.env.local の Git 管理除外
.gitignore に .env.local・.env.*.local・.env を明記。API キーがリポジトリにコミットされない。
⚠️
【今後の強化推奨】レート制限のサーバーレス対応
現在のレート制限はインメモリ実装。Vercel のサーバーレス環境ではインスタンスごとにメモリが独立するため、厳密な制限は保証されない。本番でより厳格な制限が必要な場合は Upstash Redis + @upstash/ratelimit の導入を推奨。
⚠️
【今後の強化推奨】Supabase RLS ポリシー設定
現在の公開 API は SERVICE_ROLE_KEY を使用（RLS バイパス）。SELECT するカラムを明示的に制限することで対処済み。今後 anon key + RLS ポリシーへ移行することでより安全な設計になる。
